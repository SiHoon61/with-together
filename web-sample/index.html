Product Flow

문서의 API 흐름을 그대로 살린 3단계 진입 구조

초대 링크 진입 시 사용자가 헷갈리지 않도록 최초 참가와 재참가를 명확히 나누고, 방 생성 직후에는 복구 키 저장을 가장 강하게 강조합니다.

Step 01

리더는 방과 첫 퀘스트 분위기를 함께 만든다

방 이름, 리더 닉네임, 최종 목표만 입력하면 바로 시작됩니다. 설정 부담을 낮추고, 퀘스트 예시 카드를 함께 배치해 서비스의 쓰임새를 첫 진입에서 이해하게 만듭니다.

  • `POST /v1/rooms` 결과를 바로 저장하는 흐름
  • 생성 직후 `sessionToken` 로컬 저장 유도
  • 리더용 초대 링크와 복구 키를 같은 문맥에서 노출
추천 퀘스트
Step 02

초대 링크에서는 방 분위기와 오늘 루틴을 먼저 보여준다

`GET /v1/invites/{inviteToken}` 응답을 활용해 방 이름, 최종 목표, 반복 퀘스트를 먼저 보여주고 그 다음 행동을 고르게 합니다. 사용자는 참여 전에 이 방이 자신과 맞는지 바로 판단할 수 있습니다.

  • 최초 참가와 재참가를 같은 화면에서 구분
  • 닉네임 입력 맥락을 방 소개 아래에 자연스럽게 배치
  • 퀘스트 미리보기로 참여 전 기대치 형성

Step 03

복구는 부가 기능이 아니라 핵심 안전장치처럼 보여준다

이 서비스는 회원가입이 없기 때문에 복구 키가 실질적인 계정 역할을 합니다. 그래서 설정 화면 깊숙이 숨기지 않고, 재참가 플로우와 대시보드 둘 다에서 쉽게 닿게 만드는 것이 중요합니다.

  • `POST /v1/invites/{inviteToken}/recover` 전용 입력 화면
  • 복구 키는 시각적으로 코드처럼 강조
  • 현재 세션과 별개로 새 세션이 발급된다는 점을 안내
Recovery Only

새 브라우저에서 기존 멤버 정체성을 다시 연결합니다.

왜 필요한가요?

같은 멤버라도 브라우저가 바뀌면 새 `sessionToken` 이 발급되기 때문입니다.

Design Direction

이 서비스에 맞게 의도적으로 잡은 UI 원칙

제품형 랜딩

예쁜 소개 페이지보다 실제 사용 장면을 크게 노출해서, 첫 방문자가 서비스 구조를 바로 이해하도록 설계했습니다.

복구 키의 승격

회원가입이 없으므로 복구 키는 설정 메뉴의 부속 기능이 아니라 메인 경험의 일부처럼 다뤄야 합니다.

체크의 속도

퀘스트 완료는 하루에도 자주 일어나는 행동이므로, 버튼 하나로 토글되는 촉감과 상태 대비를 강하게 줬습니다.

리더와 멤버 공존

리더 전용 관리 화면을 완전히 분리하기보다 같은 대시보드 안에서 운영 정보와 개인 액션이 함께 보이도록 배치했습니다.
